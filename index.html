Double Spend Attack

Attempts to withdraw funds a second time using the same secret. The nullifier registry should catch it.

Recipient Hijack Attack

Intercepts a valid proof and modifies the public recipient address to divert funds to an attacker's address. Schnorr signature should break.

Fake Secret Attack

Tries to claim a commitment leaf with a randomized fake secret. The commitment recalculation should fail verification.

Fake Merkle Path Attack

Submits a modified Merkle path to claim a commitment that was never deposited. Root reconstruction should mismatch.

Deep Cryptographic Insights

In our system, when a user deposits, they submit $C = \text{SHA256}(Y || n)$ where $Y = g^s \pmod p$. During withdrawal, a real zk-SNARK would require the user to send only a cryptographic proof $\pi = (\pi_A, \pi_B, \pi_C)$ containing three elliptic curve points, alongside public inputs: Merkle Root $R$, Nullifier Hash $H_n$, and recipient $A$. The circuit verifies that the prover knows the secret keys without leaking them. Our simulator models this circuit execution directly: the prover's witness values ($s$, $n$, path) are processed in the `verify_proof` function, mimicking the constraints a SNARK circuit checks, while demonstrating a real Schnorr mathematical proof for the ownership element.
